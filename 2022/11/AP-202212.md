  SI SM 프로젝트 수행 당시 기획자가 없기 때문에 클라이언트와 직접 컨택하여 업무를 수행한 경험이 많다. 처음에는 상대방이 왜 저런 생각을 했는지 이해가 가지 않아 트러블이 생겼지만, 침착하고 열린 마음으로 상대방과 소통을 하다보니 현상이나 기능을 바라보는 시각, 용어의 의미등이 서로 다름을 알게 되었고, 상대의 시각을 이해하려 하다보니 소통의 중요함을 깨닳았다.
  이런 경험으로 아이디룩에서 개발이 필요한 이슈가 생기면 기획자, 디자이너, 실사용자와 소통하여 사용자의 경험을 고려한 프로그램의 구조를 설계할 수 있게 기획에도 직접 참여한다. 소통을 통하여 사용자가 원하는 기능에 최적의 기술과 UX를 고려하여 프로젝트를 진행하였다. 결과적으로 입사 전에는 1점대의 아이디룩몰 앱(웹뷰방식) 평가를 평균 4점대로 올렸으며, 사용자의 사이트 채류 시간과 재방문율, 구매율을 얼마정도 올렸다.

  효율을 좋아하기 때문에 비는 시간에 주도적으로 소스의 리팩토링을 한다. 중복된 기능을 하는 파편화된 소스와 가독성 개선, UX 개선을 위한 공통 모듈을 개발/리팩토링하여 같이 일하는 개발자들에게 공유, 개발업무의 효율 상승을 추구한다. 대표적으로는 [아이디룩몰], [아페쎄 몰], [해양경찰청 오염방제 통합 시스템]이 있다. 해당 실적으로 불필요한 리소스를 제거하고 가독성 향상으로 유지보수 작업의 효율성을 향상시켰다.

## 직무기술서 (지원 직무와 관련하여 주요 수행업무 및 실적을 기준으로 상세히 기술해 주십시오) (3000 자)

```txt
  JD + 경력 + 성과
    핵심 경력 사항 최신순
    성과에 대한 객관적 자료
    뭐를 개발하였으며 어떻게 개선 되었다
    업무 강점 및 경험에서 얻은 전문가적 견해
    텍스트와 이미지
```

저는 현재 아이디룩의 이커머스 개발팀 주임으로 근무하며 자사 쇼핑몰(아이디룩몰, 아페쎄몰) 백엔드 개발실무를 맞고 있습니다. 상세 업무는 쇼핑몰 웹 서비스의 오류수정, 기능개선, MD의 운영지원, 통계성 데이터 추출, 장애 대응 및 해결, CS/배송 업무처리 기술 지원 그리고 인프라 관리를 하고 있습니다.

```txt
  자사 물류창고에서 배송 시작까지 3 ~ 5일 걸리던 것을 3자 물류를 연계한 것이 2차 개선  
  2차 개선시에 자사 물류창고의 부담이 줄어 2 ~ 3일이면 배송이 시작되었다.
  그러나 2차 개선시 3자 물류사와 API 협의가 잘 되지 않아 결품 처리나 합포장에 이슈가 생겼고, 이를 개선중이다.
  10월 초에는 운영자가 이벤트 진행하였고 이로인해 마일리지 지급을 1만여명에게 하려 시도 하였으나 오류로 지급하지 못하게 되었음. 오류의 원인을 찾아보니 Oracle DB 11G 상으로 1천건 이상의 값을 처리 할 수 없어 오류가 발생하였고 기존 마일리지 지급 쿼리는 IN 절에 회원 ID를 담고 있었음. 이를 개선하다 확인해보니 다른 기능 역시 다량의 데이터 DML을 IN 절에 담고 있거나 insert 또는 update를 한 건씩 트랜잭션을 물고 있어 대용량 데이터 DML 문에 성능이 굉장히 낮았음. 해당 문제를 개선하고자 MyBatis Execute Type BATCH 속성으로 대용량 데이터 처리 속도개선 하였고, Batch DML 또한 건수에 제한이 있는 것을 확인하여 목록을 페이징 처리, 1페이지가 끝날 때 마다 select sysdate를 실행하여 트랜잭션을 끝내도록 기능 개선하였음. 해당 기능 개선으로 마일리지 지급, 쿠폰생성 등의 기능의 소요시간을 1/4로 줄이는 성과를 이뤘음.
  지난 9월에는 이벤트/프로모션 기능 개선을 하였음
  기존 소스는 쿠폰 다운로드, 사다리타기, 응모, 랜덤 추첨 등의 프로모션을 한번에 한개의 이벤트만 생성하여 사용 가능하도록 개발되었고, 마케터나 운영 MD의 기획으로 이런 기능을 다중으로 사용하는 컨텐츠를 진행하게 되면 3일에서 길게는 10일의 개발 기간이 필요했음. 이로 인해 리뷰하는 시간과 디테일을 서로 조율하는 일정등으로 개발, 마케팅, MD 운영 세개 팀에서 소비되는 공수가 많다고 판단하였고 통합이벤트 기능을 제안하였음. 주도적으로 2주간의 기능 개선작업을 수행하였고, 다중 콘텐츠 프로모션을 진행하는데 소비되는 공수가 없도록 개선하였음.
  2022년 상반기에는 IDC에서 AWS로 인프라를 전환하였음.
  해당 프로젝트에서 Redis로 어플리케이션의 세션 클러스팅을 구축, OS File System 및 static resources의 S3 Cloud Front API로 전환, Tmax WebtoB에서 Nginx로 WEB SERVER 전환을 맡아 작업하였음.
  어플리케이션을 IDC에서 운영하던 당시에는 2대의 서버에서 구동하였지만 세션 클러스터링 기능을 사용하지 않아 종종 세션이 끊기는 등의 문제가 있었음. 해당 문제로 사용자는 로그인이 끊기는 등의 말도 안되는 UX를 경험하였고, 이와 관련된 CS 인입이 많았었음. 클라우드 환경으로 전환을 하면서 레디스에 세션 클러스터링을 구축하여 세션과 관련된 CS은 사라졌음.
  또한 멀티파트 파일 업로드시 OS File System을 이용하여 파일을 저장하던 어플리케이션을 AWS S3 API를 이용하도록 전환하였고, 
  static resource 들을 웹서버에서 직접 호스트 하는 시스템 구조를 Jenkins에서 빌드시 S3에 FTP로 저장하도록 설정, Cloud Front에서 호스트 하도록 개선하였음. 이 개선 작업으로 6000원/1Mbps 종량제로 IDC에서 운영하던 CDN 서비스를 3600원/1Mbps 의 AWS Cloud Front 서비스로 전환하여 월 평균 160 만원 정도의 비용을 절감하였음.
```
