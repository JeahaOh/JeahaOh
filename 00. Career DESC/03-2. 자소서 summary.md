# 자기소개

## 성격상의 장단점

저의 장점은 차분한 성격과 다른 사람을 존중하고 공감을 잘하며, 환경 적응력이 뛰어난 편이라 어디든 잘 적응하곤 합니다. 하지만 다른 사람들을 배려하고 존중하는 저의 장점은 때로는 제게 독으로 작용해 다른 사람의 부탁을 거절하지 못해 낭패를 볼 때도 있었습니다.  
프로젝트 오픈이 얼마 남지 않은 상황에서 다른 팀원을 도와주게 되어 제가 맡은 부분에 시간을 들이지 못하게 되는 상황이 있었습니다. 결국, 저는 프로젝트 완성을 위해 하루에 서너 시간 잠을 자며 작업에 몰두하였고, 프로젝트 오픈일을 맞출 수 있었습니다. 그러나 제가 맡은 부분에는 온전히 힘을 쓰지 못해 만족도가 떨어졌습니다. 이를 개선하기 위해 공과 사를 구별하고, 맺고 끊음에 있어 정확하려고 노력하고 있습니다.

## 업무 스타일이 서로 다른 구성원들과 협력해서 과제를 완수했던 경험.

현재 유지보수 중인 서비스의 이벤트 페이지에는 응모, 쿠폰, 출석 체크 등의 콘텐츠를 한 번에 한 가지의 기능만 포함할 수 있었습니다. 큰 이벤트를 진행하기 위해서는 항상 추가적인 개발 공수가 필요했습니다.  
이벤트를 기획하는 마케팅팀과 운영팀은 협업에 필요한 일정보다는 트렌디하고 빈번한 프로모션을 통해 이벤트의 효과를 증대하고 싶어 했습니다. 이벤트 기능을 담당하던 개발팀의 전임자는 기능을 모듈화하여 코드의 재사용성을 높이는 개발 방법보다는 하드 코드를 추가하여 요구사항이 들어올 때마다 추가 개발을 하는 스타일로 과업을 수행하던 사람이었습니다.  
저는 이런 스타일의 업무가 개발팀 입장에서는 소스의 파편화와 중복된 로직으로 인한 유지보수성 저해의 원인이라고 생각했습니다. 또한 마케터의 입장에서는 트렌드를 더 민첩하게 반영하지 못해 아쉬워하는 입장을 알게 되었습니다.  
프로모션의 프로세스를 개선하고자 저는 주도적으로 관련 직군의 동료들과 적극적인 소통을 진행했습니다. 처음부터 개발에 초점을 두지 않고 동료들이 필요로 하는 기능과 그 목적을 파악하기 위해 노력했습니다. 설령 구현이 불가능한 부분이 있더라도 상대방이 이런 기능을 원하는 이유에 대해 공감하도록 노력했습니다.  
저의 이런 적극적인 소통과 협력을 통해 개선된 이벤트 기능으로 마케팅 직군의 동료들은 더 유연하게 이벤트를 기획, 진행하였고 디자인 팀과 개발팀의 공수를 줄여 효율적이고 민첩하게 프로모션을 진행할 수 있게 되었습니다.

## 경력 중심 소개.

1. 회원 도메인에 대한 이해  
커머스 도메인에는 상품/전시, 마케팅, 주문/결제/CS, 배송, 회원계 등 세분된 도메인이 있고 이에 따른 다양한 기능이 필요합니다. 저는 여러 도메인중 마케팅, 배송, 회원의 고도화 및 운영 경험으로 필요한 도메인 지식을 쌓아 왔습니다.  
  
2. 자바와 스프링에 대한 높은 이해도  
저는 자바와 스프링 프레임워크를 이용한 웹 애플리케이션 운영 및 고도화를 중심으로 커리어를 만들어 왔습니다. 경험해 온 프로젝트들을 기반으로 현재 직면한 문제 파악과 적절한 기술을 적용해 문제를 해결하는 다양한 경험을 했습니다.  
  
3. DB 관리 및 SQL 작성 능력  
커머스 시스템의 전시, 검색은 대량의 데이터를 다루기 때문에 데이터베이스에 대한 이해와 쿼리 작성 능력이 중요합니다. 저는 RDBMS 설계 및 최적화 그리고 복잡한 쿼리를 작성할 수 있는 능력을 보유하고 있습니다. 최근 상품 목록 페이지의 쿼리 개선을 위해 인덱스 개선, 조회 조건 최적화, 서브쿼리 최적화, 뷰와 엠뷰의 활용, 캐싱 활용 등의 다양한 최적화 방법으로 개선했습니다.

## 대표적인 프로젝트 세 가지와 사용 기술, 구체적인 개발 방법.

### 1. 이벤트 상품 목록 조회 성능 고도화

서비스에서 수용할 수 있는 동시 접속자수를 2배로 늘린 경험이 있습니다.  
프로모션 문자 발송 시 동시 접속자 수가 약 2,000명이 넘을 경우 DB CPU 사용률이 100%가 되어 접속 장애가 발생하는 일이 생겼습니다. 이런 현상이 빈번하게 발생하기 시작했고 문제 해결을 위해 여러 방법을 모색했습니다. 이벤트 상품의 목록 조회 쿼리를 개선하는 방법, 이벤트 상품 목록을 검색엔진에 저장하여 불러오는 방법, 이벤트 상품 목록을 Redis에 저장 후 조회하는 방법 등의 해결책들이 제시되었고 여러 가지 상황을 종합한 결과 현 상황에서는 Redis를 이용하는 방법이 최선이라 판단하였습니다.  
사용할 기술이 정해지고 기능 개선을 위한 개발을 하려 보니 Java 1.7, Spring 3.1.1의 사양과 절차 지향적으로 작성된 코드 등의 기술 부채와 도메인 모델 대신 Hash Map으로 개발된 상황에서는 Annotation을 이용한 Cache 사용이 어렵다는 것을 알게 되었습니다.  
저는 Spring Data Redis와 jedis client를 직접 이용하여 Redis에 데이터를 삽입하고 조회하는 Repository 객체를 직접 구현했습니다. RedisTemplate 객체를 Bean으로 등록하여 Repository에서 사용하도록 구현했습니다.  
Repository 객체 이용하여 Scheduler 프로그램에서 20분마다 Write Around 패턴으로 상품 목록을 Caching 하였고, 상품 목록을 조회하는 컨트롤러에서는 Look Aside 패턴으로 데이터가 필요할 때 캐시에 있는 데이터가 있는지 우선 확인, 없다면 DB를 조회하는 전략으로 캐시 처리 했습니다.  
개발을 끝내고 결과적으로 마케팅 문자를 발송했을 때 약 4,000명의 동시 접속자를 받으면서 DB의 CPU 사용률을 50~60%의 안정적인 모습을 보였고 접속 장애는 더 이상 발생하지 않게 되었습니다.  



### 2. 배송 상품 목록 배송 추적 기능의 고도화

대량의 데이터 처리 시간을 약 60% 감소시킨 경험이 있습니다.  
온라인 몰에서 출고하는 주문 물량이 많아지면서 하루 평균 1,800 여 건의 배송을 추적하던 프로그램이 약 3,000건의 배송을 추적하게 되었습니다. 물류센터 및 각 매장에서 발송하는 물류는 여러 택배사를 이용했습니다. 택배사별로 배송 상태 조회 API의 응답 값 구조가 다르기 때문에 스윗트래커 API를 이용했습니다. 해당 API는 송장 번호 건별로 조회하기 때문에 상당한 자원을 소모했습니다. 3,000건의 배송을 추적하게 되며 해당 프로그램의 소요 시간이 1시간 이상 걸렸습니다.  

저는 배송추적 프로세스의 개선을 다음과 같은 순서로 작동하도록 수정했습니다.  
1. 배송추적 대상 송장 번호를 모두 DB에서 조회한다.
2. 50개씩 쪼개어 배송추적 메소드를 비동기 멀티스레드로 실행한다.
3. 응답받은 결과를 DB의 값과 비교, 상태 변경이 있는 송장 번호들만 DB에 일괄 업데이트.

이 과정에서 비동기 멀티스레드를 이용하기 위해 Async Annotation을 사용하였고, 프로그램 성능 저하 방지와 추후 다른 기능에서도 비동기 스레드를 사용하기 위해 Thread Pool을 항상 살아 있는 최소 스레드 수 10, 최대 스레드 수 50, 스레드 풀에서 사용할 최대 큐의 크기를 200으로 설정했습니다.  
또한 DB의 여러 row에 대한 업데이트를 일괄 처리 하기 위해 MyBatis ExcuteType BATCH 설정을 이용하였고, Transactional Annotation의 REQUIRED_NEW 속성 사용 시 예외 처리가 잘 되어있지 않으면 부모 트랜잭션까지 Rollback 되는 문제가 발생하여, 부모/자식 프로세스 간 트랜잭션을 독립적으로 처리 되도록 예외 처리 했습니다.

### 3. AWS 전환 (S3, Cloud Front, Jenkins 배포)

인프라를 AWS로 전환하는 프로젝트에서 다음과 같은 파트를 담당하게 되었습니다.
- 수기 배포에서 Jenkins를 활용한 배포 자동화 전환.
- IDC CDN 서비스에서 AWS Cloud Front로 전환.
- 파일 관리를 OS File System에서 AWS S3로 전환.
- Spring Framework와 Redis를 연동하여 Session Clustering 구성.

이전의 배포는 다음과 같은 절차로 진행되었습니다.
1. 배포 담당자가 이클립스에서 프로젝트를 빌드
2. WEB 서버 종료
3. ftp로 WAS 서버에 파일 덮어쓰기
4. WAS 재시작
5. WEB 서버 재시작
6. 4대의 서버에 2~5번 과정 반복

해당 과정을 통해 배포할 경우 휴먼 에러가 발생하는 경우와 사용자의 Session이 끊기는 불편한 UX의 문제가 있었습니다. 이를 개선하기 위해 저는 Jenkins로 Build 셋팅을 하였습니다. 빌드 후 static 파일을 S3의 public bucket에 전송하여 Cloud Front로 delivery 할 수 있도록 설정하였습니다. 해당 개선으로 1시간 이상 소요되던 배포 작업을 버튼 한번 클릭으로 10분 이내에 종료되도록 개선했습니다.  
또한 배포 중 세션이 끊기는 문제를 해결하기 위해 AWS ElastiCache의 Redis를 Session Clustering으로 구성하여 UX를 개선했습니다.  
관리자 페이지와 사용자가 작성하는 상품 후기 이미지 등의 첨부 파일 또한 OS File System에서 AWS S3 SDK를 이용하여 S3에 저장, Cloud Front를 이용하도록 구성했습니다. 이 과정에서 절차 지향적으로 작성되어 파편화된 파일 관리 소스를 파일관리 클래스로 분리하여 공통 클래스로 리팩토링하여 유지보수성을 향상했습니다.

### 지금까지 힘들었던 프로젝트(개발 경험)와 그 이슈를 어떻게 해결했는지 기술하시오.

3자 물류 API와의 연계를 위해 Spring 프로젝트에 Quartz로 스케줄링하여 재고관리, 출고/반품의 배송 신청/확정 관리기능의 개발이 가장 어려웠던 프로젝트입니다. 스케줄링 프로젝트가 자바 1.7로 개발되어 Stream이나 List의 filter, find 메소드를 사용할 수 없었고 해당 메소드가 필요할 때마다 반복문으로 만들어 구현해야 했습니다. 그리고 자사 상품의 상품 체계와 3자 물류의 상품 체계가 많이 달라 어려움을 겪었습니다.  
자사는 일반 상품의 상품 코드를 고유의 8자리 코드를 사용하고, 상품 코드 + 색상 + 싸이즈의 12자리 코드를 상품 품번으로 관리합니다. 자사의 세트 상품은 일반 상품을 임의로 묶어서 세트를 구성하도록 개발되었습니다. 세트 상품은 여러 개의 구성 상품 코드를 ','로 붙여 상품 코드를 구성하고, 세트의 상품품번 또한 '상품 품번,상품 품번'으로 관리합니다. 3자 물류는 상품 품번을 SKU 단위로 사용하여 상품을 관리합니다.  
해당 프로젝트 중 가장 복잡했던 로직은 반품 상품의 정상/불량의 구분하여 처리하는 기능입니다. 반송 받은 상품의 품번과 수를 반품 주문의 품번과 수와 비교 검증하는 로직을 만드는데 고생했습니다. 처음에는 간단할 것으로 생각했으나 개발 후 실제 주문과 상품을 참고해 테스트하니 사용자들은 세트 상품을 구매할 때 세트 상품의 구성 상품을 일반 상품으로 더 주문하는 케이스가 있고, 단순 비교 방식은 세트 상품 + 일반 상품 주문의 케이스에서 상품의 정상/불량 여부를 구분하는 데 문제가 있었습니다. 이에 반송 받은 데이터와 반품 주문 데이터를 {구성 상품품번 : 총 갯수} 형태로 풀어 비교 검증하는 로직으로 재개발하여 해결했습니다.  
해당 기능 개발로 기능을 설계하는 단계에서 실제 사용자의 사용 케이스와 데이터에 대한 분석과 파악을 더 꼼꼼하게 하게 되는 계기가 되었습니다.
