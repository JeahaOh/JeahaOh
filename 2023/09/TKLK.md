### 자신에 대해 경력 중심으로 자유롭게 표현해 보세요. (경력 중심) (필수)

저는 적극적인 소통능력을 이용하여 성과를 이뤄내는 것을 좋아합니다. 빠른 문제 파악 및 업무처리에는 좋은 소통이 필요하다고 생각합니다. 큰 성과는 혼자서 하는 것이 아니라 협업을 통해 여러 사람의 노력이 필요한 만큼 다양한 직무의 팀원과 협업에 가치를 두고 업무에 임합니다. 여러 직군의 동료들과 적극적인 커뮤니케이션을 통해 협업하고 개발을 진행하여 서비스의 품질과 비즈니스 로직 그리고 사용자의 UX를 개선한 경험이 있습니다.  
기존의 이벤트/프로모션 페이지에서는 한 번에 한 개의 콘텐츠(응모, 쿠폰, 랜덤, 출석 체크 등)만 포함할 수 있었고 마케팅 직군의 동료들은 더 풍성하고 빈번한 콘텐츠를 통해 프로모션의 효과를 증대하고 싶어 했습니다. 한 이벤트에서 여러 콘텐츠를 함께 진행하기 위해서는 항상 추가적인 개발 공수가 필요했고 주기적인 프로모션과 반복적인 추가 개발은 민감한 트렌드를 따르지 못하는 문제도 있었습니다.  
이에 저는 언제나 여러 콘텐츠의 이벤트를 게시할 수 있도록 주도적으로 문제를 해결하고 싶었습니다. 커뮤니케이션하기 위해 직군 간 표현을 크로스 체크하였고, 실제로 해결하고 싶은 문제를 해결하기 위해 충분한 소통을 하였습니다. 이슈 사항에 대해 상대방이 이해하기 어려운 기술적인 용어 사용과 장황한 설명을 지양했습니다. 처음부터 개발에 초점을 두지 않고 해당 기능이 필요하게 된 계기나 기능으로 기대할 수 있는 상대방의 의도와 목적을 파악하고, 설령 구현이 불가능한 부분이 있더라도 상대방이 이런 기능을 원하는 이유에 대해 공감하도록 노력했습니다.  
저의 이런 적극적인 커뮤니케이션을 통해 개선된 이벤트/프로모션 기능으로 마케팅 직군의 동료들은 더 유연하게 이벤트를 기획, 진행하였고 디자인 팀과 개발팀의 공수를 줄여 효율적이고 민첩하게 프로모션을 진행할 수 있게 되었습니다. 저의 소통에 대한 가치관과 능력으로 귀사에서 의미있는 새로운 문제에 도전하고 더 좋은 세상을 만드는데 기여하고 싶습니다.

### 대표적인 프로젝트 세 가지를 적어 주시되, 각각의 프로젝트에서 어떤 기술을 활용하여 어떤 방식으로 개발했는지 구체적으로 설명해 주세요 (필수)

#### 이벤트 상품 목록 조회 성능 고도화

프로모션 오픈안내 마케팅 문자 발송시 일정 수준의 접속자 수가 넘을 경우 DB CPU가 100%를 치고 접속 장애가 발생하는 일이 생겼습니다. 이런 현상이 빈번하게 발생하기 시작했고 문제 해결을 위해 여러 방법을 모색했습니다. 이벤트 상품의 목록 조회 쿼리를 개선하는 방법, 이벤트 상품 목록을 검색엔진에 저장하여 불러오는 방법, 이벤트 상품 목록을 Redis에 저장후 조회하는 방법 등의 해결책들이 제시되었고 여러가지 상황을 종합한 결과 현 상황에서는 Redis를 이용하는 방법이 최선이라 판단하였습니다.  
사용할 기술이 정해지고 기능 개선을 위한 개발을 하려 보니 Java 1.7, Spring 3.1.1의 사양과 절차지향적으로 작성된 코드, 기술부채로 인해 도메인 모뎅 대신 HashMap으로 개발된 상황에서는 Annotation을 이용한 Cache 사용이 어렵다는 것을 알게 되었습니다.  
저는 Spring Data Redis와 jedis client를 직접 이용하여 Redis에 데이터를 삽입하고 조회하는 Repository 객체를 직접 구현했습니다. RedisTemplate 객체를 Bean으로 등록하여 Repository에서 사용하도록 하니 구현이 어렵지는 않았습니다.  
Scheduler 프로그램에서 20분마자 Write Around 패턴으로 상품 목록을 Caching 하였고, 상품 목록을 조회하는 컨트롤러에서는 Look Aside 패턴으로 데이터가 필요할 때 캐시에 있는 데이터가 있는지 우선 확인, 없다면 DB를 조회하는 전략으로 캐시처리를 했습니다.  
개발을 끝내고 결과적으로 마케팅 문자를 발송했을 때 약 1000 명의 동시 접속자를 더 받으면서 DB의 CPU 사용율을 60~60%의 안정적인 모습을 보였고 접속 장애는 더 이상 발생하지 않게 되었습니다.

#### 배송 상품 목록 배송 추적 기능의 고도화

대량의 데이터 처리 시간을 약 60% 감소시킨 경험이 있습니다.  
온라인 몰에서 출고하는 주문 물량이 많아지면서 하루 평균 1800 여 건의 배송을 추적하던 프로그램이 약 3000건의 배송을 추적하게 되었습니다. 물류센터 및 각 매장에서 발송하는 물류는 여러 택배사를 이용했습니다. 택배사별로 배송상태 조회 API의 응답 값 구조가 다르기 때문에 스윗트래커 API를 이용했고 송장번호 건별로 조회하기 때문에 프로그램의 종료까지 걸리는데에는 상당한 시간이 걸렸습니다. 3000건의 배송을 추적하게 되며 해당 프로그램의 소요시간은 1시간이 넘어 버리게 됬습니다.

저는 배송추적 프로세스를 개선을 다음과 같은 순서로 작동하도록 수정했습니다.
1. 배송추적 대상 송장번호를 모두 DB에서 조회한다.
2. 50개씩 쪼개어 배송추적 메소드를 비동기 멀티쓰레드로 실행한다.
3. 응답 받은 결과를 DB의 값과 비교, 상태 변경이 있는 송장 번호들만 DB에 일괄 업데이트.

이 과정에서 비동기 멀티쓰레드를 이용하기 위해 Async Annotation을 사용하였고, 프로그램 성능 저하 방지와 추후 다른 기능에서도 비동기 쓰레드를 사용하기 위해 Thread Pool을 항상 살아 있는 최소 쓰레드 수 10, 최대 쓰레드 수 50, 스레드 풀에서 사용할 최대 큐의 크기를 200으로 설정 했습니다.  
또한 DB의 여러 row에 대한 업데이트를 일괄 처리 하기 위해 MyBatis ExcuteType BATCH 설정을 이용하였고, Transactional Annotation의 REQUIRED_NEW 속성 사용시 예외처리가 잘 되어있지 않으면 부모 트랜잭션까지 Rollback 되는 문제가 발생하여, 부모/자식 프로세스 간 트랜잭션을 독립적으로 처리 되도록 예외처리 했습니다.

#### AWS 전환 (S3, Cloud Front, Jenkins 배포)

인프라를 AWS로 전환하는 프로젝트에서 다음과 같은 파트를 담당하게 되었습니다.
- 수기 배포에서 Jenkins를 활용한 배포 자동화 전환.
- IDC CDN 서비스에서 AWS Cloud Front로 전환.
- 파일 관리를 OS File System에서 AWS S3로 전환.
- Spring Framework와 Redis를 연동하여 Session Clustering 구성.

이전의 배포는 다음과 같은 절차로 진행되었습니다.
1. 배포 담당자가 이클립스에서 프로젝트를 빌드
2. WEB 서버 종료
3. ftp로 WAS 서버에 파일 덮어쓰기
4. WAS 재시작
5. WEB 서버 재시작
6. 4대의 서버에 2 ~ 5번 과정 반복

해당 과정을 통해 배포를 할 경우 휴먼 에러가 발생하는 경우와 사용자의 Session이 끊기는 불편한 UX의 문제가 있었습니다. 이를 개선하기 위해 AWS의 Code Deploy와 각 인스턴스를 Blue/Green 배포 설정을 선임이 담당하여 설정하였고, 저는 Jenkins로 Build 셋팅을 하였습니다. 빌드 후 static 파일을 S3의 public bucket에 전송하여 cloud front로 delivery 할 수 있도록 설정하였습니다. 해당 개선으로 2시간 이상 소요되던 배포작업을 버튼 한번 클릭으로 30분 이내에 종료되도록 개선했습니다.  
또한 배포 중 세션이 끊기는 문제를 해결하기 위해 Redis를 Session Clustering으로 구성하여 UX를 개선했습니다.  
어드민페이지와 사용자가 작성하는 상품 후기 이미지 등의 첨부 파일 또한 OS File System에서 AWS S3 SDK를 이용하여 S3에 저장, Cloud Front를 이용하도록 구성했습니다. 이 과정에서 절차지향적으로 작성되어 파편화된 파일 관리 소스를 파일관리 클래스로 분리하여 공통 클래스로 리팩토링하여 유지보수성을 향상 시켰습니다.
