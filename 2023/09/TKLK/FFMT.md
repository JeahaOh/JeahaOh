3자 물류 API와의 연계를 Spring 프로젝트에 Quartz로 스케쥴링하여 기능 자동화. (재고관리, 결품관리, 출고/반품 배송 신청/실적 관리 기능)
해당 프로젝트에서 가장 복잡했던 로직은 반품완료 처리기능의 응답받은 반품확정실적수신 데이터에서 값을 분류하는 부분이 제일 어려웠다.
그 이유는 자사 데이터구조는 셋트상품일 경우 세트상품의 키를 따로 만들어 하나의 상품으로 관리하지만, 3자 물류측에서는 단일상품의 상품코드로 상품을 관리하기 때문에 구성상품의 상품 코드를 풀어서 요청, 응답해야 하기 때문이다.

반품완료 처리기능의 실행시 비지니스 로직은 다음 수도코드와 같이 작동했다.
```
전송대상_조회()

3자물류_전용_API_클라이언트.반품확정실적수신()

수신결과_실패
  종료;

응답_데이터_분류()

반품상테변경데이터.주문정보.반품배송상태_수정()
반품수거완료데이터.주문정보.반품완료_수정()
반품불가데이터.기록저장()
```

응답_데이터_분류의 수도코드는 다음과 같다.

```
  for 응답_받은_상품_리스트


```

단일상품일 경우 반송 받은 상품의 정상상품 갯수와 사용자에게 반품 요청받은 상품의 갯수를 비교하는 7줄이면 충분했다.
