# 이벤트 상품 목록 조회 성능 고도화

- 2023.07.24 ~ 2023.08.17
- Redis를 사용하여 이벤트 페이지 하위 상품 목록의 조회성능 개선.
- DB CPU 사용율 감소와 그에 따른 소화 가능 트래픽 증가.

프로모션 오픈 마케팅 문자 발송시 일정 수준(얼마나?)의 접속자 수가 넘을 경우 DB의 CPU가 100%를 차고 접속 장애가 발생함.  
이런 현상이 빈번하게 발생하기 시작했고, 문제해결을 위해 다양한 방법을 모색함.  
- 이벤트 페이지를 미리 html로 만들어 마켓팅 문자를 보내는 방법.
- 이벤트 상품 목록 조회의 쿼리를 개선하는 방법.
- 이벤트 상품 목록을 검색엔진에 저장하여 불러오는 방법.
- 이벤트 상품 목록을 Redis에 저장한 후 조회하는 방법.
등의 방법 들이 거론 됨.  
여러 가지 상황을 종합한 결과 현 상황에서 Redis를 이용하는 방법으로 해결하기로 결정됨.  
사용할 기술이 정해지고 개발을 하려보니 java 1.7, spring 3.1.1의 사양과 절차지향적으로 작성된 코드, 기술부채로 인해 도메인 모델 대신 HashMap으로 개발된 상황에서는 Annotation을 이용한 Cache 사용이 어렵다는 것을 파악하게 되었음.  
SpringDataRedis와 jedis client를 직접 이용하여 Redis에 데이터를 삽입하고 조회하는 Repository 객체를 만들기로 결정하게 됨.  
RedisTemplate 객체를 Bean으로 등록하여 Repository에서 사용하도록 하니 구현이 어렵지 않았음.  
Scheduler 프로그램에서 20분 마다 Write Around 패텅으로 상품 목록을 Caching 하였고, 상품 목록을 조회하는 곳에서 Look Aside 패턴으로 데이터가 필요할 때 캐시에 있는 데이터가 있는지 우선 확인, 없다면 DB를 조회하는 전략으로 캐싱처리를 하였음.  
개발을 끝내고 결과적으로 마케팅 문자를 발송했을 때 약 1000명의 동시 접속자를 더 받으면서 DB의 CPU 사용율을 60~70%의 안정적인 모습을 보였고 접속 장애는 더 이상 발생하지 않았음.

# 이벤트 상품 목록 조회 성능 고도화

프로모션 오픈안내 마케팅 문자 발송시 일정 수준의 접속자 수가 넘을 경우 DB CPU가 100%를 치고 접속 장애가 발생하는 일이 생겼습니다. 이런 현상이 빈번하게 발생하기 시작했고 문제 해결을 위해 여러 방법을 모색했습니다. 이벤트 상품의 목록 조회 쿼리를 개선하는 방법, 이벤트 상품 목록을 검색엔진에 저장하여 불러오는 방법, 이벤트 상품 목록을 Redis에 저장후 조회하는 방법 등의 해결책들이 제시되었고 여러가지 상황을 종합한 결과 현 상황에서는 Redis를 이용하는 방법이 최선이라 판단하였습니다.  
사용할 기술이 정해지고 기능 개선을 위한 개발을 하려 보니 Java 1.7, Spring 3.1.1의 사양과 절차지향적으로 작성된 코드, 기술부채로 인해 도메인 모뎅 대신 HashMap으로 개발된 상황에서는 Annotation을 이용한 Cache 사용이 어렵다는 것을 알게 되었습니다.  
저는 Spring Data Redis와 jedis client를 직접 이용하여 Redis에 데이터를 삽입하고 조회하는 Repository 객체를 직접 구현했습니다. RedisTemplate 객체를 Bean으로 등록하여 Repository에서 사용하도록 하니 구현이 어렵지는 않았습니다.  
Scheduler 프로그램에서 20분마자 Write Around 패턴으로 상품 목록을 Caching 하였고, 상품 목록을 조회하는 컨트롤러에서는 Look Aside 패턴으로 데이터가 필요할 때 캐시에 있는 데이터가 있는지 우선 확인, 없다면 DB를 조회하는 전략으로 캐시처리를 했습니다.  
개발을 끝내고 결과적으로 마케팅 문자를 발송했을 때 약 1000 명의 동시 접속자를 더 받으면서 DB의 CPU 사용율을 60~60%의 안정적인 모습을 보였고 접속 장애는 더 이상 발생하지 않게 되었습니다.
