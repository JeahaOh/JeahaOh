## 1. MVC 1, MVC 2의 차이

MCV란 SW 아키텍쳐 디자인 패턴 중 하나로 데이터를 갖는 Model, 사용자 화면을 담당하는 View, 비즈니스 로직을 처리하는 Controller로 이루어진 패턴이다.  
  
### MVC 1

뷰와 컨트롤러의 경계가 없음. 따라서 비즈니스 로직을 담당하는 Java 코드와 UI/UX를 담당하는 HTML, CSS, JavsScript가 섞여있음.  
코드가 복잡해지며 유지보수가 힘들어짐.  
개발 속도가 빠르다는 장점 외에는 기술 부채를 만들어 내는 주체로 볼 수 있음.

### MVC 2

유지보수성을 보완하기위해 나옴.  
비즈니스 로직과 UI/UX가 분리되어 확장이 용이하고 유지보수성이 높아짐.
초기 설계 단계에 비용이 조금 더 들수도 있음.

사족으로는 MVC2가 이미 익숙하기 때문에 개발 속도가 느리다는 것에 동의 할 수 없고, MVC2역시 3-tier 계층 구조를 잘 따르지 않으면 유지보수가 어려워진다고 생각하고 있습니다.

## 4. 검색이 어떤 방식으로 작동하는가?

(포털의 검색엔진인지, 사이트 내 검색 엔진인지?)  
기본적인 검색엔진은 역색인 방식으로 동작한다. 예를 들면 책의 가장 마지막에 있는 찾아보기 페이지와 같다.  
검색 엔진은 문장을 공백 단위로 쪼개 역색인 작업을 한다.  
A에 '나는 책을 읽는다'에 '나는 영화를 본다'라는 문장을 대입 했을 때 '나는'이라는 토큰은 A, B에 모두 있고, '책을', '영화를', '본다', '읽는다'의 토큰은 A와 B에 각각 있다.  
검색엔진은 이런 token, id 형식의 구조로 저장한다. '영화' 라는 문장이 검색엔진에게 요청되면 B 값을 응답하는 게 역색인 방식을 사용하는 검색엔진의 원리이다.  

포털의 검색엔진은 크롤러가 URL을 따라 웹 페이지들을 이동한다. 출발점이 되는 URL은 포털에 등록 신청된 웹사이트, 오픈디렉토리, ISP에서 제공하는 서버 목록등에서 가져온다.  
방문한 웹 페이지에서 단어와 문구를 분리 저장, 각 단어와 문구에 가중치와 연관도를 부여하고 색인을 한다.  
검색자가 검색어를 입력하면, 그 때 필요한 것을 바로 꺼내는 것이 아니라 검색엔진이 미리 만든 색인을 조회하여 연관도와 가중치에 따라 결과를 보여준다.

## 5. 대용량 데이터 처리를 위한 서비스 아키텍쳐에 대한 설명과 기술

MSA는 웹 기반의 분산 시스템의 디자인으로 특정 사람이 정의한 아키텍쳐는 아니였으나, 마틴 파울러가 MSA에 대한 개념을 글로 정리하며 개념을 정립했다.  
MSA나 모노리스틱 구조에서 어느 한쪽이 우세하게 장/단점을 가지고 있진 않고, 상황에 맞는 구성을 해야한다.

### 모노리스틱 아키텍쳐

모노리스틱 아키텍쳐는 하나의 어플리케이션 내에 모든 로직들이 들어 있는 구조이다. 사용자 관리, 상품, 전시, 주문 등과 UX/UI까지 하나로 포장되어 있다. 이런 구조는 규모가 큰 애플리케이션에서 크기 때문에 빌드 및 배포, 기동까지 시간이 오래 걸리며 한두사람의 실수로 전체 시스템에 오류를 일으킬 수 있기 때문에 프로젝트의 규모가 커질 수록 협업도 쉽지 않다. 그 외에 구조적으로 대규모 시스템에 사용하기에는 단점이 많다.

## Micro Service Architecutre

MSA는 대용량 서비스가 많아짐에 따라 정의된 아키텍쳐다. MSA는 사용자 관리, 상품, 전시, 주문 등과 UX/UI의 각 컴포넌트를 서비스하는 개념이다. 각 컴포넌트를 수직적으로 분할하여 REST API와 같은 표준 인터페이스로 기능을 제공한다.  

더 상세한 내용이 있지만, 직접 사용해 보지 않아서 개념이 머릿속에 남지 않았다.

## 8. 정렬 알고리즘 중 가장 빠른 것과 그 설명

가장 빠른 정렬 알고리즘은 퀵정렬로 Divide And Conquer 기법을 통해 정렬하는 방식이다.  

1. 주어진 배열에서 임의의 한 원소를 피벗으로 고른다.
2. 피벗을 기준으로 피벗보다 값이 작은 원소들을 모은 배열과 큰 값의 원소들을 모은 배열로 나눈다.
3. 분할된 두 배열에 대해 1, 2 과정을 재귀적으로 반복한다.

- 장점 :  
    - 웬만 해선 O(nlogn)의 시간복잡도를 갖는다.
    - 추가적인 메모리를 공간을 필요로 하지 않는다.
- 단점 :  
    - 불안정한 알고리즘이다
    - 이미 정렬된 상태에서 수행할 경우 O(N^2)까지 성능이 떨어진다.

## 11. RESTfull API는 무엇인가?

### 목적

- 이해하기 쉽고 사용하기 쉬운 API를 만드는 것.

### REST

- Representational State Transfer의 약자로 자원을 이름으로 구분하여 해당 자원의 정보를 주고 받는 것을 의미함.
- JSON 혹은 XML을 통해 데이터를 주고 받음.
- HTTP 프로토콜을 그대로 사용하기 때문에 웹의 장점을 최대한 활용할 수 있는 아키텍쳐 스타일.
- HTTP URI를 통해 자원을 명시하고, HTTP METHOD(GET, POST, PUT, DELETE)를 통해 해당 자원에 대한 CRUD Operation을 적용하는 것을 의미함.

### 구성요소

- 자원 (URI)  
  클라이언트는 URI를 이용해 자원을 지정하고 해당 자원에 대한 조작을 요청할 수 있음.
- 행위 (Verb) :  
  - HTTP Method 이용
  - GET POST PUT DELETE
- 표현 (Representation Of Resource)

### 장점

- 별도의 인프라를 구축할 필요가 없음.
- HTTP 프로토콜을 사용하는 모든 플랫폼에서 사용 가능.
- 여러 서비스 디자인에 생길 수 있는 문제를 간소화 함.
- 서버와 클라이언트의 역할을 명확하게 분리함.

### 단점

- 표준이 없음
- 사용할 수 있는 메소드가 4가지 밖에 없음. (GET, POST, PUT, DELETE)

### 필요성

- 애플리케이션의 분리 및 통합.
- 다양한 클라이언트의 등장. (브라우져, 모바일 앱)
- 멀티 플렛폼에 대한 지원을 위한 아키텍쳐.

### 설계 규칙

1. 슬래쉬(/)는 계층관계를 나타냄
2. URI 마지막에는 슬래시를 포함하지 않음.
3. 가독성을 위해
    - 대쉬(-)를 사용함
    - 언더스코어(_)는 문자가 가려지기도 하므로 사용하지 않음
4. URI 경로는 소문자를 사용
5. 파일 확장자는 URI에 포함하지 않음.
    - header의 Accept에 이미 명시됨.
6. 리소스 간에 연관 관계가 있는 경우
    - `/리소스명/리소스 ID/관계가 있는 다른 리소스명`
    - 일반적으로 'has' 관계를 표현함.

## 12. HTTP 상태 코드

- 1XX : 프로토콜 수준의 정보 교환
- 2XX : 클라이언트 요청이 수행됨
    - **200 OK** : 성공적 처리
- 3XX : 클라이언트의 요청을 완료하기 위해 추가적인 행동이 필요함
    - 302 Found : 일시적으로 컨텐츠가 이동됨.
    - 303 See Other : 영구적으로 컨텐츠가 이동됨.
    - **304 Not Modified** : 변경 없음 -> Cache된 정보를 사용하라
    - 307, 308 : 302와 같다.
- 4XX : 클라이언트의 잘못된 요청
    - 400 Bad Request : 요청 자체가 잘못 됨
    - 401 Unauthorized : 권한 없음.
    - **403** Forbidden : 거부됨.
    - **404 Not Found** : 찾을 수 없음.
    - **405** Method Not Allowed : 허용되지 않은 메소드 요청시.
    - 406 Not Acceptable : 웹 방화벽등 서버에서 받아 들일 수 없는 요청 일 경우.
    - 408 Request Timeout : 요청 중 시간이 초과되었음.
    - 413 Payload Too Large : 요청 본문이 긴 경우.
    - **415 Unsupported Media Type** : 요청한 타입이 서버에서 지원하지 않는 경우.
- 5XX : 서버측 오류
  - 500 Internal Server Error : 처리중 오류 발생시.
  - 502 Bad Gateway : 게이트웨이가 연결된 서버로부터 잘못된 응답을 받았을 경우.
  - 503 Service Temporarily Unavailable : 과부하등의 문제로 일시적으로 응답 불가.
  - 504 Gateway Timeout : 게이트웨이가 연결된 서버로부터 응답을 받을 수 없음.

## 20. POJO란?

Plain Old Java Object의 약자로, 객체지향적인 원리에 충실하면서, 환경과 기술에 종속되지 않고, 필요에 따라 대활용 될 수 있는 방식으로 설계된 객체.
