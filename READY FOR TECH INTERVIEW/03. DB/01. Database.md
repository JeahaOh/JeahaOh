# Database

## DB의 특징

1. 데이터의 독립성
    - 물리적 독립성 :  
      DB의 사이즈를 늘리거나 성능 향상을 위해 데이터 파일을 늘리거나 새롭게 추가하더라도 관련된 응용 프로그램을 수정할 필요가 없다.
    - 논리적 독립성 :  
      DB는 논리적인 구조로 다양한 응용프로그램의 논리적 요구를 만족시켜 줄 수 있다.
2. 데이터의 무결성
    여러 경로를 통해 잘못된 데이터가 발생하는 경우의 수를 방지하는 기능으로 데이터의 유효성 검사를 통해 데이터의 무결성을 구현한다.
3. 데이터의 보안성
    인가된 사용자들만 DB나 DB 내의 자원에 접근할 수 있도록 계정 관리 또는 접근 권한을 설정하여 보안을 구현할 수 있다.
4. 데이터의 일관성
    연관된 정보는 논리적인 구조로 관리함으로 어떤 하나의 데이터만 변경했을 경우 발생할 수 있는 데이터의 불일치성을 배제할 수 있다. 또한, 작업 중 일부 데이터만 변경되어 나머지 데이터와 일치하지 않는 경우의 수를 배제할 수 있다.
5. 데이터 중복 최소화
    DB는 데이터를 통합해서 관리함으로써 파일시스템의 단점 중 하나인 자료의 중복과 데이터의 중복성 문제를 해결할 수 있다.

## INDEX

책의 맨 처음과 마지막에 있는 색인과 같다. 데이터가 저장된 레코드의 주소는 인덱스 목록에 있는 페이지 번호가 된다.  
항상 정렬된 상태를 유지하기 때문에 원하는 값을 탐색하는 데는 빠르지만, 새로운 값을 추가하거나 삭제/수정하는 경우에는 속도가 느려진다. 조회하는 데 사용하는 컬럼이라고 모두 인덱스로 생성하면 **데이터 저장 성능이 떨어지고 인덱스의 크기가 비대해져 역효과**만 불러온다.  
결론적으로 인덱스는 저장 성능을 희생하고 읽기 속도를 높이는 기능이다.

### B Tree

- 이진 탐색트리와 유사
- 자식 노드를 둘 이상 가질 수 있고, Balanced Tree라는 특징이 있다.
- 탐색 연산에 있어 O(log N)의 시간 복잡도를 갖는다.
- 모든 노드가 값의 저장과 포인터 역할을 동시에 한다.

### B+ Tree

- B Tree의 변형 구조
- 값을 리프노드에만 저장하며 리프 노드들 끼리는 링크드리스트로 연결되어 있다.
- 리프노드를 제외한 노드들은 포인터의 역할만 수행한다.
- 장점
  - 블럭 사이즈를 더 많이 이용 가능
  - 리프 노드들끼리 링크드 리스트로 연결되어 범위 탐색에 유리
- 단점
  - B Tree의 경우 최상의 경우 루트에서 조회가 끝날 수 있지만, B+ Tree는 무조건 리프 노드까지 내려가 봐야 한다.

## 정규화

```txt
  정규형은 너무 이론적으로만 길고, 이해하기는 어렵다. 예시 설계를 직접 해봐야 알 것 같다.
```

- 테이블 간에 중복된 데이터를 허용하지 않는 것.
- 이상 현상이 있는 릴레이션을 분해하여 이상 현상을 없애는 과정. 정규형이 높아질수록 이상 현상은 줄어듦.
- 무결성을 유지하며 DB의 저장공간을 줄일 수 있다.
- 장점
  - 이상 현상을 제거할 수 있다.
  - 새로운 데이터형의 추가로 인한 확장 시, 그 구조를 변경하지 않아도 되거나 일부만 변경해도 된다.
  - 연동된 응용 프로그램에 사이드이펙트를 최소화할 수 있다.
- 단점
  - 릴레이션 분해로 인해 JOIN 연산이 많아진다.
  - query에 대한 응답이 느려질 수 있다.
  - 조인이 많아 발생하는 성능저하는 반정규화를 적용할 수도 있다.

### 이상현상

- 삽입이상 (Insertion Anomaly) : 특정 값에 해당하는 값이 없어 NULL을 입력해야 하는 현상
- 삭제이상 (Deleteion Anomaly) : 튜플 삭제 시 같이 저장된 다른 정보까지 연쇄적으로 삭제되는 현상
- 갱신이상 (Update Anomaly) : 튜플 갱신 시 중복된 데이터의 일부만 갱신되어 일어나는 데이터 불일치 현상

### 함수 종속성 (FD : Functional Dependency)

- 함수 종속성은 어떤 속성 A의 값을 알면, 다른 속성 B의 값이 유일하게 정해지는 관계를 종속성이라고 한다.
- A -> B로 표현하면 
- A -> B
    - A를 B의 결정자(Determinant)라고 한다.
    - A는 B를 결정(Determine)한다.
    - B는 A에 종속한다(Dependent)고 한다.

### 1 정규형 (1NF)

1. 각 컬럼이 하나의 속성만을 가져야 한다.
2. 하나의 컬럼은 같은 종류나 타입(type)의 값을 가져야 한다.
3. 각 컬럼이 유일한(unique) 이름을 가져야 한다.
4. 컬럼의 순서가 상관없어야 한다.

### 2 정규형 (2NF)

1. 1 정규형을 만족해야 한다.
2. 모든 컬럼이 부분적 종속(Partial Dependency)이 없어야 한다.
    - 모든 컬럼이 완전 함수 종속을 만족해야 한다.

### 3 정규형 (3NF)

1. 2 정규형을 만족해야 한다.
2. 기본키를 제외하나 속성들 간의 이행 종속성(Transitive Dependency)이 없어야 한다.

### BCNF (Boyce-Codd Normal Form)

1. 3 정규형을 만족해야 한다.
2. 모든 결정자가 후보키 집합에 속해야 한다.

## Transaction

**작업의 완정성을 보장**하는 것. 

### TX와 Lock

lock은 동시성을 제어하기 위한 기능. TX는 데이터의 정합성을 보장하기 위한 기능.  
lock은 여러 커넥션에서 동시에 동일한 자원을 요청할 경우 순서대로 한 시점에는 하나의 커넥션만 변경할 수 있게 해주는 역할.

### TX의 4요소 (ACID 특성)

- Atomicity 원자성 :  
    TX가 모두 반영되거나 전혀 반영되지 않아야 한다. All Or Nothing
- Consistency 일관성 :  
    작업 처리 결과는 항상 일관성이 있어야 한다.
- Isolation 고립성(독립성) :  
    둘 이상의 TX가 동시에 실행될 때, 어떤 TX도 다른 TX의 연산에 끼어들 수 없다.
- Durability 지속성 :  
    TX가 성공적으로 완료되면, 결과는 영구적으로 반영되어야 함.

### Commit, Rollback

- Commit
    하나의 TX가 성공적으로 끝났고, DB가 일관성있는 상태일 때 이를 알려주기 위해 사용하는 연산.
- Rollback
    하나의 TX가 비정상 종료되어 원자성이 깨진 경우, TX의 시작 상태로 돌아가기 위한 연산.

### Dead Lock (교착상태)

복수의 TX가 특정 자원(테이블 또는 행)의 lock을 획득한 채, 다른 TX에서 소유하고 있는 lock을 요구하면, 아무리 기다려도 상황이 바뀌지 않는 상태.
